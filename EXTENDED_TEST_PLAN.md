# План розширеного тестування проекту ReadAloud

Цей документ визначає стратегію та конкретні тестові сценарії для заповнення прогалин у тестовому покритті додатка, зокрема в частині прикладних сервісів, інтеграції компонентів, багатопотоковості та інтерфейсу користувача.

## 1. Unit-тести прикладних сервісів (`src/application/services`)

Ці тести мають фокусуватися на логіці управління станами та взаємодії між доменами.

### 1.1 QueueService
- `test_add_task_sequence`: Перевірка коректної нумерації та черговості доданих завдань.
- `test_status_transition_validation`: Валідація дозволених переходів станів (наприклад, перевірка неможливості переходу з `COMPLETED` назад у `PENDING`).
- `test_subscriber_notifications`: Перевірка, що всі підписані компоненти (Controller, UI) отримують відповідні сигнали при зміні стану черги.

### 1.2 GenerationService
- `test_batch_submission_logic`: Перевірка розбиття великої кількості чанків на батчі згідно з параметром `BATCH_GEN_SIZE`.
- `test_generation_retry_logic`: Симуляція помилок мережі/API та перевірка коректності роботи механізму повторних спроб (RetryHandler).
- `test_pause_resume_behavior`: Перевірка зупинки активних робочих потоків при команді Pause та їх коректне відновлення без дублювання запитів.

### 1.3 AssemblyService
- `test_batch_completion_tracking`: Перевірка логіки очікування всіх частин (parts) перед початком фінальної збірки.
- `test_assembly_strategy_selection`: Автоматичний вибір між стратегіями `fast` (копіювання кодека) та `full` (перекодування) залежно від встановленої швидкості відтворення.
- `test_cleanup_after_assembly`: Перевірка видалення тимчасових файлів `part_X.mp3` після успішного склеювання фінального файлу.

## 2. Інтеграційні тести (Integration layer)

Перевірка "ланцюжків" компонентів без глибокого використання моків для внутрішньої логіки.

- `test_text_to_filesystem_integration`: Повний цикл: Raw Text -> `TextProcessor` (очищення) -> `TextChunker` (розбиття) -> `FileManager` (запис на диск). Перевірка цілісності даних на кожному етапі.
- `test_app_orchestration_integration`: Перевірка взаємодії `AppController` з `GenerationService` та `AssemblyService` на мінімальному наборі даних для підтвердження правильності передачі сигналів та параметрів.

## 3. Тестування конкурентності (Concurrency & Stress)

Перевірка стабільності системи під навантаженням та в умовах багатопотоковості.

- `test_concurrent_queue_management`: Одночасне додавання та видалення великої кількості завдань з різних потоків для виявлення потенційних race conditions у `QueueService`.
- `test_rapid_cancel_during_processing`: Тест на "живучість": скасування завдання саме в момент активної генерації або збірки файлів для перевірки коректного звільнення ресурсів.

## 4. UI-тестування (`src/gui`)

Фокус на валідації вводу та реакції інтерфейсу на зміни стану системи.

- `test_project_name_validation`: Перевірка роботи валідатора назви проекту (заборона спецсимволів, довжина, порожні значення).
- `test_file_path_existence_validation`: Перевірка реакції UI на вибір неіснуючого файлу або файлу без прав доступу на читання.
- `test_dynamic_ui_updates`: Перевірка синхронізації стану кнопок (Start/Stop/Pause) із реальним станом черги завдань.
